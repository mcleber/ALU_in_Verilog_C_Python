# Part 1 — Fundamentals

## Chapter 1 — Environment setup

### 1.1 Tools used

We will use the following tools:

```text
Icarus Verilog -> compile and simulate Verilog code
vvp            -> run the simulation generated by Icarus
GTKWave        -> visualize signals as waveforms
GCC            -> compile C code
Python 3       -> automate tests and generate vectors
VSCode         -> editing and debugging environment
Make           -> automate repetitive commands
```

### 1.2 Installation on Linux or WSL

```bash
sudo apt update
sudo apt install iverilog gtkwave gcc make gdb python3 python3-venv python3-pip
```

Verify that everything is installed:

```text
iverilog -V
vvp -V
gcc --version
python3 --version
make --version
```

### 1.3 Recommended VSCode extensions

Install:

- C/C++;
- Python;
- Verilog/SystemVerilog;
- Makefile Tools, optional.

## Chapter 2 — The role of each language

### 2.1 Verilog

Verilog describes **hardware**.

When we write:

```verilog
assign y = a & b;
```

we are not saying:

```text
Execute an AND instruction.
```

We are saying:

```text
There is an AND logic gate connecting a and b to y.
```

This difference is fundamental.

In C and Python, the program executes line by line. In Verilog, several blocks operate at the same time, like real circuits.

### 2.2 C

C will be used as the **reference model**.

This means we will implement the same ALU in C, in a simple way, so we know what result the hardware should produce.

For example:

```text
Input: A = 10, B = 3, operation = ADD
Expected result: 13
```

C will be our "teacher" telling us the correct result.

### 2.3 Python

Python will be used for **automation**.

It will:

- compile the C program;
- run unit tests in C;
- generate test vectors;
- compile Verilog;
- run the simulation;
- compare the expected result with the obtained result;
- show a final report.

### 2.4 Why integrate the three languages?

Because this flow resembles real projects.

In professional hardware projects, it is common to have:

```text
Reference model          -> C, C++, Python or MATLAB
Hardware                 -> Verilog, SystemVerilog or VHDL
Verification automation  -> Python, Make, scripts, CI
```

In this guide, we will build a small, educational version of that.

## Chapter 3 — Essential concepts of digital systems

### 3.1 Bit

A bit can be:

```text
0 or 1
```

In digital hardware, all data is made of bits.

### 3.2 Bit vector

An 8-bit vector can represent values from 0 to 255:

```text
00000000 = 0
00000001 = 1
00000010 = 2
11111111 = 255
```

In Verilog:

```verilog
wire [7:0] data;
```

This means:

```text
data has 8 bits: data[7] down to data[0]
```

### 3.3 Combinational logic

Combinational logic is logic in which the output depends only on the current inputs.

Example:

```text
y = a AND b
```

If `a` or `b` changes, `y` changes shortly after.

A simple ALU is mostly combinational.

### 3.4 Sequential logic

Sequential logic depends on a clock and memory.

Example:

```text
register
counter
FSM
pipeline
```

The output may depend on the previous state.

### 3.5 Clock

The clock is a periodic signal.

```text
0 -> 1 -> 0 -> 1 -> 0 -> 1
```

Many circuits update their registers on the rising edge:

```verilog
always @(posedge clk)
```

### 3.6 Reset

Reset puts the circuit into a known state.

Without reset, an FSM may start in an undefined state.

### 3.7 Automated testing

An automated test is a test that reports by itself whether it passed or failed.

Bad:

```text
Run the simulation and look at the waveform manually.
```

Good:

```text
Run the simulation and the testbench prints PASS or FAIL.
```

In this guide, we will always aim for self-checking tests.
